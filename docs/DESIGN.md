# DESIGN.md

`multi_path_peer_review` エージェントで確立した 5 つの設計原則。各原則は実装の反復を通じて発見・articulation されたものであり、別プロジェクトでも適用可能な汎用性を持つ。

## 用語

本ドキュメントが参照する設計概念:

- **制約の 4 段階モデル**: LLM 出力の構造的制約を 4 階層で設計するアプローチ
  1. プロンプト（自然言語での指示・規範）
  2. Pydantic / 構造化出力（型レベル制約）
  3. コードループ上限（`max_iterations` 等）
  4. コードフィルタ（score 閾値等）
- **多層防御 (defense-in-depth)**: 上記 4 段階を組み合わせ、いずれか単独の段階が破綻しても全体が暴走しない設計姿勢

## 原則 1: peer-review × grounding の効果

各 path が独立に外部 grounded (Tavily) な状態で他視点と相互参照することで、内部閉じた self-reflection / cross-reflection より強い fact verification 効果を生む。self_reflection は同じ LLM が同じ誤認を通し、cross_reflection は別 LLM でも元の数値根拠を持っていなければ判定不能。本実装は **各 path が独立に grounding を持つ** という性質が決定的に作用し、「自分は X、他視点は Y、Tavily で再検証する」という構造が成立する。

**臨床エビデンス**: smoke run で **数値桁ミス** が Round 2 で検出・修正された事例: Round 1 で大規模モデルのパラメータ数を「1.3 億」と記述（正しくは 1.3 billion = 13 億、billion ↔ 億の単位換算ミス）。Round 2 で他視点との数値矛盾として検出され、Tavily 再検証で訂正された。self/cross-reflection では原理的に発見不可能。

**適用条件**: 各 path が独立に grounding を持つこと、N >= 3 推奨。視点間に意味的交差がない場合は効果が出ない。

## 原則 2: 定数の責務分離

ライフサイクル段階ごとに同じ意味の定数も独立に定義する。現状の値が同じでも、意味が異なるなら別定数として扱う。

**具体例**:

```python
MAX_ITERATIONS_PER_PATH = 2     # Round 1 のリサーチ用途 Tavily 上限
MAX_ITERATIONS_ROUND2 = 2       # Round 2 の矛盾解消用途 Tavily 上限
```

両者は現状で同じ値だが意味は別。「現状値が同じだから 1 つにまとめる」最適化は、後の責務変更で必ず破綻する: 片方の責務だけ変えたいときに両方の挙動を変える必要が出て、その時点で別定数化しても git history に「なぜ分けたか」が残らず設計判断が暗黙化する。最初から別定数として定義することで、設計意図がコードレベルで明示される。

## 原則 3: ライフサイクル段階に応じた制約強度の非対称設計

制約の 4 段階モデルにおける各段階の **強度** (Pydantic max_length 等) は、ライフサイクル段階の責務に応じて非対称に設計する。

**具体例**: `section_markdown` の max_length:

| 段階 | 場所 | max_length | 責務 |
|---|---|---|---|
| Round 1 LLM 出力 | `_PathLLMOutput` | 1,700 | 過剰生成抑制 (strict) |
| Round 2 LLM 出力 | `_PathRound2LLMOutput` | 2,500 | 改訂機会保護 (relaxed) |
| 公開データ型 | `PathResult` | 2,500 | 両 Round の最大値に揃える |

Round 1 は初稿執筆 (Sonnet の包括的傾向を抑制)、Round 2 は改訂・cross-reference・空白補填 (文字数増が本質的に必要)。同じ制約を課すと Round 2 の改訂機会が構造的に剥奪され、権威ソースで裏付けた事実修正が fallback で破棄される事象が観察された。

**defense-in-depth の根拠**: Round 1 の strict gate (max=1,700) が下流のベースラインを保証するため、Round 2 の relaxation を許しても多層防御は破綻しない。Round 1 と Round 2 は独立ではなく階層的依存関係にある。

**臨床エビデンス**: 開発過程で `PathResult.max_length` を Round 2 と同期し忘れた際、LLM が Round 2 で 2,500 字以内を生成しても PathResult 構築時に max=1,700 で reject されて ValidationError → graph crash が発生。同じ意味のフィールドが複数の型に存在する場合、その制約は同期管理する必要がある。

## 原則 4: 規範ベース response 制御

LLM の挙動を抑制する手段として、ハード制約 (Pydantic max_length 等) ではなく **prompt 規範** で意思決定起点に介入する設計を優先する。

**背景: constant chasing 問題**
LLM (特に Sonnet) は構造的限界点まで使う傾向がある。max=1,700 にすると 1,700 ぎりぎりまで書き、緩めて 2,500 にすると 2,500 まで書く。`target` や `推奨` のような soft 指示では「使ってよい上限」と解釈され抑制されない。

**規範ベース制御の設計**: prompt で意思決定の起点を再定義する。

1. **本質再定義**: 「Round 2 はゼロから書き直すのではなく、Round 1 への最小限の差分」
2. **絶対基準ではなく相対基準**: 「target 1,300-1,600 字」ではなく「Round 1 ± 200 字」
3. **上限の意味の明示**: 「max_length は目標値ではなく異常時の防御である」

**効果**: 規範導入前の ai_tech Round 2 は max=2,500 超過 → 導入後は **1,665 字に収束** (max の 67%)。4 視点平均は Round 1 ± 236 字、**fallback firing 0 件**。ハード制約だけでは捕捉しきれない LLM の挙動を、prompt 規範で構造的限界点に到達する前に抑制できた。

規範ベース制御は多層防御の **段階 1 (プロンプト)** を強化することで、段階 2 (Pydantic) への依存を減らす。両者は対立せず補完関係にある。

## 原則 5: 視点間整合性のための粒度設計

複数 path が parallel に同じ事実を扱うとき、より **粗い粒度** で表現することで自然な整合性が得られる。精度を下げることで信頼性を上げるエンジニアリングのトレードオフ。

**臨床エビデンス**: smoke run で **日付の日単位ズレ** が同一レポート内で発生する事象を観察: 同じイベント (AI 気象モデルの運用開始日) について 4 視点が独立に Tavily で日付を確認し、わずかに異なる情報源から「12 月 17 日」「12 月 18 日」「12 月中」の 3 通りを引いた。日単位で正しく取れる情報源が存在するからこそ、視点間で食い違いが発生する。

**解決策**: prompt で日付のデフォルト粒度を **月単位** に統一する。例外として日単位を許容するのは論文発表日・公式文書の発行日・特定イベント日のみ。

**副次効果**: その日付情報そのものが本文から消失する事例も観察。LLM が「日付確定が困難な場合は概念レベルで書く」方向に動いた。粗い粒度を選ぶと情報量が減るトレードオフは構造的に残る。

## 原則間の関係

5 つの原則は独立ではなく相互に関係する:

```
原則 1: peer-review × grounding (本実装の核心)
        ↓
原則 2: 定数の責務分離  ←→  原則 3: 制約強度の非対称設計
   (責務分離を定数レベルで)    (同じ姿勢を Pydantic max_length に)
        ↓
原則 4: 規範ベース response 制御
   (ハード制約に頼らず prompt 規範で LLM 制御)
        ↓
原則 5: 粒度設計 (規範ベース制御の特殊形、cross-perspective 整合性向上)
```

## まとめ

- 原則 1 が **実装の核心** (なぜ peer-review か)
- 原則 2-3 が **責務分離の articulation** (定数 → Pydantic スキーマ)
- 原則 4-5 が **response 制御の articulation** (規範ベース → 粒度設計)

規範ベース response 制御は、ハード制約への依存を減らし LLM の自律的抑制を引き出す **設計手法の核心** であり、文字数だけでなく表記・時間・粒度といった他領域にも展開可能な汎用性を持つ。
