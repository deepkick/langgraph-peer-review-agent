# DESIGN.md

`multi_path_peer_review` エージェントで確立した 5 つの設計原則。

各原則は実装の反復を通じて発見・articulation されたものであり、別プロジェクトでも適用可能な汎用性を持つ。

## 用語

本ドキュメントが参照する設計概念:

- **制約の 4 段階モデル**: LLM 出力の構造的制約を 4 階層で設計するアプローチ
  1. プロンプト（自然言語での指示・規範）
  2. Pydantic / 構造化出力（型レベル制約）
  3. コードループ上限（`max_iterations` 等の構造的上限）
  4. コードフィルタ（score 閾値等のポストプロセス）
- **多層防御 (defense-in-depth)**: 上記 4 段階を組み合わせ、いずれか単独の段階が破綻しても全体が暴走しない設計姿勢

## 原則 1: peer-review × grounding の効果

### 主張

複数 path が **外部 grounded (Tavily 検索)** な状態で **他視点と相互参照** することで、内部閉じた self-reflection / cross-reflection より強い fact verification 効果を生む。

### 背景

self_reflection / cross_reflection / role_based_cooperation のような peer-review 系パターンは、いずれも LLM の出力を **LLM 自身または別 LLM が評価する** 構造であり、外部の grounded source による事実検証を伴わない。これらのアプローチには以下の限界がある:

- self_reflection: 同じ LLM が自分の出力を見直すため、同じ誤認は通る
- cross_reflection: 別 LLM が評価しても、元の数値や事実根拠を持っていなければ判定不能

本実装の peer-review は、**各 path が独立に grounding を持つ** (Tavily 検索結果を own draft の根拠としている) ため、他視点との相互参照で「自分は X と書いた、他視点は Y と書いている、どちらが正しいか Tavily で再検証する」という構造が成立する。

### 臨床エビデンス

ある smoke run で観察された Aurora パラメータ数の検出事例:

- ai_tech (Round 1): Microsoft Aurora を「1.3 億パラメータ」と記述 (誤訳、正しくは 1.3 billion = 13 億)
- Round 2 で他視点が Aurora を扱う過程で数値の桁が他視点と矛盾することに LLM が気づき、Tavily で再検証
- 修正は revision_notes に記録された

この種の誤認は self_reflection / cross_reflection では原理的に発見不可能。peer-review の **半 grounding 付き** という性質が決定的に作用する。

### 適用条件

- 各 path が独立に grounding を持つ仕組みを備えていること (本実装では Tavily)
- 複数 path が同じ事実領域に触れる可能性があること (path 間に意味的な交差がない場合は効果が出ない)
- N >= 3 推奨 (2 path だと矛盾が出ても判定材料が少ない)

### 限界

- LLM が改訂を本文に正しく反映する保証はない (revision_notes と本文の乖離が観察されることがある、原則 4 と関連)
- fallback で改訂が破棄される可能性 (原則 3 と関連)
- 視点間で事実を扱わない場合 (例: 各視点が完全に独立した側面のみ扱う) は効果が出ない

## 原則 2: 定数の責務分離

### 主張

ライフサイクル段階ごとに **同じ意味の定数でも独立に定義する**。現状の値が同じでも、意味が異なるなら別定数として扱う。

### 具体例

```python
MAX_ITERATIONS_PER_PATH = 2     # Round 1 の Tavily 上限 (path_executor)
MAX_ITERATIONS_ROUND2 = 2       # Round 2 の Tavily 上限 (path_executor_round2)
```

両者は現状で同じ値だが、意味は別:
- `MAX_ITERATIONS_PER_PATH`: Round 1 のリサーチ用途 Tavily 上限
- `MAX_ITERATIONS_ROUND2`: Round 2 の矛盾解消用途 Tavily 上限

将来、Round 2 を「矛盾解消のみだから 1 回で十分」と判断した場合、`MAX_ITERATIONS_ROUND2 = 1` に独立に変更できる。逆に Round 1 のリサーチ深度を上げたい場合、Round 1 だけ `MAX_ITERATIONS_PER_PATH = 3` にできる。

### なぜ別定数化するか

「現状値が同じだから 1 つの定数にまとめる」という最適化は、後の責務変更で必ず破綻する。具体的には:

1. 1 つの定数でまとめている状態で、片方の責務だけ変更したい時、両方の挙動を変える必要が出る
2. その時点で別定数化すると、git history で「なぜ分けたのか」が記録されない
3. 結果、「これらは同じ値であるべきか別であるべきか」の設計判断が暗黙化する

最初から別定数として定義することで、設計意図がコードレベルで明示される。

### 適用範囲

本原則は「定数」について articulate したが、より広く **同じ名前のフィールドや変数が複数の意味で使われる場合に分離する** という設計姿勢として一般化できる。原則 3 (Pydantic Field の制約) もこの姿勢の応用。

## 原則 3: ライフサイクル段階に応じた制約強度の非対称設計

### 主張

制約の 4 段階モデルにおける各段階の **強度 (Pydantic max_length 等の構造的制約値)** は、ライフサイクル段階の責務に応じて **独立に、非対称に** 設計する。

### 具体例

`section_markdown` フィールドの `max_length`:

| 段階 | 場所 | max_length | 責務 |
|---|---|---|---|
| Round 1 LLM 出力 | `_PathLLMOutput` | 1,700 | 過剰生成抑制 (strict) |
| Round 2 LLM 出力 | `_PathRound2LLMOutput` | 2,500 | 改訂機会保護 (relaxed) |
| 公開データ型 | `PathResult` | 2,500 | 両 Round の最大値に揃える |

### 設計の正当性

Round 1 と Round 2 は責務が異なる:
- Round 1: 初稿執筆、過剰生成 (Sonnet の包括的傾向) を抑制する規律が必要
- Round 2: 改訂・cross-reference・空白補填、文字数増が本質的に必要

これらに同じ制約 (max_length=1700) を課すと、Round 2 の改訂機会が構造的に剥奪される。実際、緩和前の試行では Round 2 が 1700 字超過で fallback 発火し、権威ソースで裏付けた事実修正 (AIFS-ENS の運用化時期、WeatherNext 2 のアーキテクチャ等) が完全に破棄される事象が観察された。

### defense-in-depth が成立する根拠

Round 2 を緩めても多層防御は破綻しない。理由:

- Round 1 の strict gate (max=1700) が **下流のベースライン** を保証する
- Round 1 で生成された PathResult は max=1700 内に収まっている
- Round 2 で参照する own_draft も max=1700 内
- Round 2 が relaxation を活かして 2500 字まで使えるのは、Round 1 をベースに「改訂で増える分」のみ

つまり Round 1 の strict gate は **Round 2 の relaxation の必要条件** であり、両者は独立ではなく階層的依存関係を持つ。

### PathResult.max_length の同期問題

`PathResult` は Round 1 と Round 2 双方の出力を保持する公開型。その max_length は **両 Round の最大値** (= 2,500) に揃える必要がある。

これを忘れると graph crash する。LLM が Round 2 で 2500 字以内の output を生成しても、PathResult 構築時に max=1700 で reject されて ValidationError → 致命エラー。

教訓: **同じ意味のフィールドが複数の型に存在する場合、その制約は同期して管理する必要がある**。これは原則 2 (定数の責務分離) と表裏の関係: 定数は分離すべきだが、同じフィールドの max は同期すべき (ただし「同じフィールド」の定義は型の責務次第)。

### 一般化

本原則を抽象化すると:

> 構造的制約 (Pydantic, JSON Schema 等) はライフサイクル段階の責務に応じて非対称に設計する。strict 段階が下流のベースラインを保証することで、relaxed 段階の存在が defense-in-depth を破らない。

これは Pydantic に限らず、任意のライフサイクル設計 (例: 開発環境 vs 本番環境のバリデーション、起動時 vs 実行時のチェック) に適用可能。

## 原則 4: 規範ベース response 制御

### 主張

LLM の挙動を抑制する手段として、**ハード制約 (Pydantic max_length 等) ではなく prompt 規範 (norm)** で意思決定起点に介入する設計を優先する。ハード制約は constant chasing を生み、規範指示は LLM の自律的な抑制を引き出す。

### 背景: constant chasing 問題

LLM (特に Sonnet) は **構造的限界点まで使う傾向** がある:
- max_length=1700 にすると 1700 ぎりぎりまで書く
- 緩めて max_length=2500 にすると 2500 ぎりぎりまで書く
- さらに緩めて max_length=3000 にすると 3000 まで書く

この挙動は `target` や `推奨` のような soft 指示では抑制されない。「使ってよい上限」と解釈される。constant を chase してもキリがない (上限を上げ続けても暴走が続く事例を観察済み)。

### 規範ベース制御の設計

prompt で **意思決定の起点** を再定義する。具体的には:

1. **本質再定義**: 「Round 2 はゼロから書き直すのではなく、Round 1 への最小限の差分を生成する」と明示
2. **絶対基準ではなく相対基準**: 「target 1,300-1,600 字」ではなく「Round 1 ± 200 字」のように、起点となる draft からの差分で表現
3. **上限の意味の明示**: 「max_length は目標値ではなく異常時の防御である」と明記

### 効果

規範ベース prompt を導入した結果:

- ai_tech Round 2: 規範導入前は 2500 字超過 → 導入後は 1,665 字に収束 (max=2500 の 67%)
- 4 視点平均: Round 1 ± 236 字 (完全な ± 200 内ではないが、絶対値暴走は完全解消)
- fallback firing: 0 件

ハード制約だけでは捕捉しきれない LLM の挙動を、prompt 規範で **構造的限界点に到達する前に** 抑制できた。

### 多領域への展開

規範ベース制御は文字数だけでなく、複数領域に展開可能:

| 領域 | 規範の例 |
|---|---|
| 文字数 | 「Round 1 ± 200 字」「最小限の差分」 |
| 表記 | 「視点参照は日本語ラベルのみ、内部 ID を本文に出さない」 |
| 時間 | 「現在は 2026 年、2023-2026 を同等の重要度で扱う」 |
| 粒度 | 「日付は月単位 default、日単位は論文発表日のみ」(原則 5) |

### 多層防御との関係

規範ベース制御は多層防御の **段階 1 (プロンプト)** を強化することで、段階 2 (Pydantic) への依存を減らす。両者は対立するのではなく補完関係:

- 段階 1 (規範): LLM の意思決定に介入、自律的抑制を引き出す
- 段階 2 (Pydantic): 異常時の構造的防御、暴走を防ぐ

両方を持つことで、通常時は規範で滑らかに動き、異常時は構造的制約で破綻を防ぐ二層構造が成立する。

## 原則 5: 視点間整合性のための粒度設計

### 主張

複数 path が parallel に同じ事実を扱うとき、より **粗い粒度** で表現することで自然な整合性が得られる。精度を下げることで信頼性を上げる、エンジニアリングのトレードオフ。

### 背景

Round 2 の各 path は parallel に実行されるため、他 path の Round 2 出力は見えない (生成中)。複数 path が同じ事実を独立に異なる粒度で扱うと、最終レポートに事実不整合が残る。

ある smoke run で観察された事例: NOAA AIGFS/AIGEFS/HGEFS の運用日が、同一レポート内で「2025年12月17日」「2025年12月18日」「2025年中」の 3 通りで混在。各視点が独立に Tavily で日付を確認し、わずかに異なる情報源 (AWS Open Data Registry vs NWS SCN 25-89) から日単位の異なる値を引いてしまった。

### 解決策: 粒度の統一

prompt で **デフォルト粒度を月単位に統一** する:
- 機関のモデル運用化、製品発表、データセット公開: 月単位 (年月)
- 例外として日単位を許容: 論文発表日、公式文書の発行日、特定のイベント日

これにより:
- 12/17 と 12/18 → どちらも「12 月」で統一
- 情報源の差が表面化しない
- ユーザーの本来の関心 (時系列の big picture) と整合

### 適用結果

粒度ルール導入後の観察:

- AIFS-ENS 運用日: 「2025 年 7 月 1 日」 → 「2025 年 7 月」で 3 視点統一
- WeatherNext 2 発表日: 「2025 年 11 月 17 日」 → 「2025 年 11 月」で 3 視点統一
- 論文発表日: Lam et al., Science 2023 のように年単位で維持

### 副次効果と限界

副次効果:
- AIGFS/AIGEFS/HGEFS の日付情報そのものが本文から消失する事例も観察 (LLM が「日付確定が困難な場合は概念レベルで書く」方向に動いた)
- 月内の表記揺れが残る (例: WMO 声明で「9 月」vs「9〜10 月」)

限界:
- 月単位でも揃わないケース (情報源で月が違う、解釈が違う) は構造的に残る
- 数値・固有名詞・パラメータ数等、日付以外の整合性問題は別途対処が必要

### 一般化

本原則は日付に限らず、**複数の独立 agent が同じ対象を扱う際の表現粒度の選択** という、より広い設計テーマの一例。

## 原則間の関係

5 つの原則は独立ではなく、相互に関係する:

```
原則 1: peer-review × grounding (本実装の核心、効果)
        ↓
原則 2: 定数の責務分離  ←→  原則 3: 制約強度の非対称設計
   (Round 1/2 の責務分離を       (同じ姿勢を Pydantic max_length に
    定数レベルで articulate)      適用)
        ↓
原則 4: 規範ベース response 制御
   (ハード制約に頼りきらず prompt 規範で LLM 制御)
        ↓
原則 5: 粒度設計
   (規範ベース制御の特殊形、cross-perspective 整合性向上)
```

原則 1 が本実装の **核心 (なぜ peer-review か)** を articulate し、原則 2-5 がその実装を機能させるための **設計手法** を articulate している。

## まとめ

本実装で確立した 5 つの設計原則:

- 原則 1: 本実装の核心 (peer-review × grounding)
- 原則 2-3: 責務分離の articulation (定数 → Pydantic スキーマ)
- 原則 4-5: response 制御の articulation (規範ベース → 粒度設計)

これらは別プロジェクトでも適用可能な汎用性を持ち、特に **規範ベース response 制御** の発見は LLM agent の prompt design における重要な転換点となる。

ハード制約だけでなく、LLM の意思決定起点に介入する **prompt 規範** によって構造的限界に到達する前に挙動を抑制する設計姿勢は、constant chasing を回避し、LLM の自律性と安全性のバランスを取るための核心的アプローチである。
